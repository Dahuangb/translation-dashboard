# 综合翻译模型评估看板 (Dashboard) 开发总结

## 1. 项目背景与目标
本项目的核心目标是构建一个专业、美观、具备权威性的“综合翻译模型评估看板”（Dashboard），用于展示和评估不同翻译模型（如 Gemini-3-Flash, GPT-4.1-mini, CLA, Gemini-3.1-Pro 等）在不同语言市场中的翻译质量和风险召回表现。

**核心要求**：
- 采用“总-分”叙事架构，首页必须高亮 SOTA（当前推荐）模型。
- 支持全局维度的宏观数据对比，以及单模型、单语言市场的微观数据下钻。
- 构建一个支持多条件组合过滤的 **Bad-case 案例检索台**，用于深入分析翻译失败的真实案例。
- 视觉风格专业、克制，统一 CSS 样式规范，避免大杂烩式的色彩和过度装饰。

---

## 2. 演进过程与核心重构

整个 Dashboard 的开发经历了从“原型堆砌”到“产品级看板”的多次重构，主要节点如下：

### 阶段一：数据接入与基础呈现
- **数据源**：最初基于一份 Word 报告的数据，人工提取成结构化的 JSON。
- **页面布局**：使用 Streamlit 的默认表格和雷达图进行展示。
- **用户反馈**：雷达图不适合超过 10 种以上的语言扩展；没有突出“总-分”逻辑，页面显得过于学术和呆板。

### 阶段二：视觉规范与图表升级
- **引入 Plotly 图表**：用分组柱状图替代原有的单调表格，加入了 `TP/FP/FN/MQM` 等核心指标的可视化对比。
- **废弃误导性图表**：将分类模型间的对比从折线图替换为分组横向条形图（小倍数图），避免制造虚假的趋势感。
- **CSS 注入与组件化**：通过 `st.markdown(unsafe_allow_html=True)` 注入全局 CSS，定义了 `hero-card`, `metric-card`, `risk-card`, `case-card` 等 5 类标准化组件。
- **统一色彩语义**：严格控制颜色数量，主品牌色（蓝）、胜出强调色（绿）、风险告警色（红橙）、常规灰阶，杜绝“调色盘打翻”的视觉感受。

### 阶段三：数据解耦与 Bad-case 检索台构建
- **解耦数据与视图**：创建 `data/translation_data.json` 存储宏观聚合指标，彻底分离了硬编码数据。
- **构建 ETL 流水线**：编写了 `parse_bad_cases.py` 脚本，用于读取各个模型输出的原始 CSV 分析报告。
- **支持多模型下钻**：逐步从单模型（Gemini-3.1-Pro）扩展到 4 个模型（加入 GPT-4.1-mini, CLA, Gemini-3.1-Flash）。脚本将分散的 CSV 统一清洗、聚合成一个 `translation_bad_cases_all_models.json` 供前端检索。

### 阶段四：架构完善与字段精准化 (Attribution)
- **缓存陷阱排查**：修复了 Streamlit 因缓存参数名带 `_` 前缀导致的缓存无法及时刷新的问题，确保新跑出的模型数据能够实时上板。
- **精准错误归因**：废弃了早期的“根据不准确理由的文本进行正则盲猜”的推断逻辑。通过解析风控裁判模型的真实输出字段 `attribution`（如 `omitted`, `mistranslated`, `sanitized`），将真实、权威的裁判结论直接作为“误译模式”的筛选项。
- **图表重叠修复**：对 Plotly 的图表边缘、Y轴高度和 Label 间距进行精细调整，解决了小倍数图模型名称截断、单语种下钻数值截断等前端展示问题。

## 3. 功能模块详解

目前的 Dashboard 被清晰地划分为以下几个核心模块：

### 1. 结论总览 (Hero 区)
- **定位**：用最快的时间告诉决策者“该用谁”。
- **展示**：左侧明显的推荐状态标签（如“推荐上线”），附带 4 张核心指标卡（综合排名、风险召回率、平均 MQM、判定精度）。

### 2. 总体表现对比
- **风险召回率**：采用分组柱状图，支持切换“综合口径 / GPT-5 口径 / Gemini 口径”，明确标注当前推荐模型的优势。
- **翻译质量 (MQM)**：采用四宫格小倍数条形图，对比各模型的 平均 MQM、TP、FN、FP。

### 3. 跨语种稳定性
- **热力图总览**：通过颜色梯度直观展现各模型在各语种的召回表现。
- **单语种下钻**：支持选择特定语言市场（默认选中差距最大的语言），通过条形图直观展示该市场下各模型与第一名的“掉队差距”。
- **稳定性辅助指标**：提出 `稳定性评分 = 100 - 召回波动`，将稳定性从主观感觉转化为可量化的证据。

### 4. 主要失误类型 (缺陷画像)
- **定位**：分析各模型的“失败倾向”。
- **展示**：顶部卡片总结每个模型的致命缺陷（如“漏翻失控”、“未翻译问题明显”），下方使用堆叠分组条形图呈现 `漏翻、未翻译、错译、和谐、捏造` 五大类错误的绝对数量分布。

### 5. 典型案例与检索台 (Master-Detail 架构)
- **单条证据卡**：默认展开一条最严重的翻车案例，清晰展示“原文风险点”、“错误译法”、“漏判原因”和“可能后果”。
- **多模型检索台**：
  - **丰富的筛选器**：支持按模型、裁判口径、风险类型、误译模式 (`attribution`)、来源字段、语言以及自由关键词组合过滤。
  - **两栏式布局 (Master-Detail)**：左侧为命中结果列表（带单选按钮），右侧为该案例的完整详情面板。这种布局避免了以往大量 Expander 堆叠带来的沉重感，大幅提升了阅卷效率。

## 4. 数据流转架构

1. **原始数据**：
   - 宏观统计数据：`translation_data.json`
   - 微观 Bad-case：散落在各处的 `translation_bad_cases_analysis_*.csv`。
2. **数据清洗 (ETL)**：
   - 执行 `python3 parse_bad_cases.py`。
   - 脚本读取所有 CSV，解析 `risk_list`、裁判模型输出结果和 `attribution` 字段。
   - 输出统一的 `data/translation_bad_cases_all_models.json`。
3. **前端渲染**：
   - `app.py` 监听 JSON 文件的修改时间（`os.path.getmtime`），触发 Streamlit 缓存刷新。
   - 前端通过 Pandas 对 JSON 数据进行过滤、分组和排序，最终交由 Plotly 和自定义 CSS 渲染输出。
